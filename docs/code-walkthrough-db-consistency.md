# Code Walkthrough: DB Consistency + Idempotency + Eventual Consistency

> Оновлено: 2026-03-22  
> Ціль: підготувати чітке пояснення для захисту, як система зберігає цілісність даних при асинхронних інтеграціях (RabbitMQ / Elasticsearch / SignalR).

---

## 1) Коротка версія (30–60 секунд)

1. Коментар спочатку синхронно пишеться в БД (`Comments`) в межах основної транзакційної операції.
2. Після успішного save запускаються асинхронні side-effects: події в MQ, realtime push, індексація в ES.
3. Якщо асинхронні інтеграції відстають або тимчасово недоступні — первинні дані в БД вже консистентні.
4. Для повторної доставки message використовується idempotency-журнал `ProcessedMessages`.
5. Це дає модель: **strong consistency для primary write** + **eventual consistency для read-моделей/інтеграцій**.

---

## 2) База даних: що гарантує цілісність

## 2.1 Таблиці

- `Comments` — головне джерело істини для коментарів і дерева відповідей (`ParentId` -> `Comments.Id`).
- `ProcessedMessages` — технічний журнал оброблених message для idempotency.

## 2.2 Інваріанти, які варто проговорити

- Коментар не «існує наполовину»: або збережений у БД, або операція не завершилась.
- Відповідь (`reply`) завжди прив’язана до валідного `ParentId` (або `null` для root).
- Повторна обробка одного й того самого message не повинна дублювати downstream-дії.

---

## 3) Транзакційна межа у create comment

## Крок A. Вхід у use-case

- API/GraphQL делегує `CreateCommentCommand` у `CreateCommentCommandHandler`.
- Валідація виконується до запису (FluentValidation через pipeline behavior).

## Крок B. Синхронний primary write

- Handler створює доменну сутність `Comment`.
- Через репозиторій/DbContext виконується запис у таблицю `Comments`.
- На цьому етапі ми отримуємо гарантовано консистентний первинний стан.

## Крок C. Асинхронні side-effects після save

- Публікується подія «коментар створено».
- Далі включається fan-out: RabbitMQ consumer-и, SignalR нотифікація, ES-індексація.
- Якщо будь-який канал тимчасово недоступний, primary write все одно залишається валідним.

---

## 4) Idempotency: як працює `ProcessedMessages`

## Проблема

У message broker допустима повторна доставка того самого повідомлення (at-least-once delivery).

## Рішення

Consumer перед виконанням перевіряє, чи `MessageId` уже є в `ProcessedMessages`:

1. Якщо запис є — обробка пропускається (duplicate).
2. Якщо запису нема — виконується бізнес-дія (наприклад, індексація), після чого `MessageId` фіксується як оброблений.

## Що сказати на захисті

> «Ми приймаємо at-least-once модель доставки і компенсуємо її на рівні обробника через idempotency store».

---

## 5) Eventual consistency: де і чому виникає

## Приклади «вікон неузгодженості»

1. Користувач створив коментар, але ES-індекс оновиться через кілька секунд.
2. Realtime-подія в один клієнт прийшла швидше, ніж у інший (мережеві відмінності).
3. MQ consumer тимчасово не працював і «наздоганяє» backlog пізніше.

## Чому це нормально

- Система спеціально розділяє **write-path** і **integration-path**, щоб API залишався швидким і стабільним.
- На read-рівні можливі короткі затримки, але функція не ламається.
- Для пошуку додатково є resilience/fallback стратегія (перехід на БД-пошук при проблемах ES).

---

## 6) Типові race-condition сценарії і пом’якшення

### Сценарій 1: duplicate delivery в MQ
- **Ризик:** подвійна індексація/подвійна побічна дія.
- **Захист:** `ProcessedMessages` + перевірка перед обробкою.

### Сценарій 2: ES недоступний під час індексації
- **Ризик:** новий коментар не одразу з’явиться в full-text пошуку.
- **Захист:** retry/pізніша обробка + fallback-пошук через repository.

### Сценарій 3: reorder подій при навантаженні
- **Ризик:** короткочасно неідеальний порядок у вторинних проєкціях.
- **Захист:** джерело істини — БД; UI може робити refetch/оновлення через стандартні query.

---

## 7) Міні-шпаргалка для усної відповіді (2–3 хв)

1. «Сильна консистентність гарантується на primary write у БД».  
2. «Усе, що стосується MQ/ES/SignalR — асинхронні side-effects із eventual consistency».  
3. «Повторні доставки повідомлень не страшні, бо є idempotency-store `ProcessedMessages`».  
4. «Якщо ES відстає/падає, пошук деградує через fallback, але функція не вимикається».  
5. «Такий дизайн зберігає і стабільність API, і масштабованість інтеграцій».
