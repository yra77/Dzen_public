# Dzen_public — walkthrough: performance, security, operations readiness

> Оновлено: 2026-03-22  
> Призначення: дати цілісний «захисний» сценарій по нефункціональних вимогах (NFR) без розпорошення по різних файлах.

---

## 1) Що саме закриває цей документ

1. **Performance**: чому система не деградує при зростанні навантаження і як контролюється latency.
2. **Security**: як захищений ввід користувача (captcha, sanitizer, валідація вкладень, error-contract).
3. **Operations readiness**: як аргументувати готовність до запуску/експлуатації (health, smoke/go-no-go, артефакти).

---

## 2) Performance — як пояснювати на захисті

## 2.1 Архітектурні рішення, що впливають на швидкодію

- **CQRS-поділ read/write**: створення коментаря і читання/пошук мають окремі use-cases і не блокують один одного.
- **Async side-effects**: індексація, messaging та realtime не «обтяжують» критичний write-path до БД.
- **Pagination by default**: root-list читається сторінками (дефолтний page size), що стабілізує обсяг відповіді.
- **Resilient search**: при проблемах Elasticsearch запит не «падає», а переходить у degraded mode через fallback.

## 2.2 Ключовий аргумент для комісії

> Ми оптимізуємо не тільки «happy path», а й поведінку при збоях: користувач отримує відповідь навіть коли частина інфраструктури тимчасово недоступна.

## 2.3 Що показати як доказ

- walkthrough data-consistency (`docs/code-walkthrough-db-consistency.md`);

## 3) Security — як пояснювати на захисті

## 3.1 Захист на вході запиту

- **Captcha**: базовий/recaptcha сценарій з challenge-перевіркою перед створенням коментаря.
- **Валідація моделей**: уніфіковано через MediatR pipeline + FluentValidation.
- **Санітизація тексту**: whitelist-підхід до XHTML/HTML-подібного вводу (щоб відсікати небезпечні конструкції).

## 3.2 Захист вкладень

- whitelist допустимих MIME-типів;
- ліміти розміру (у т.ч. окремий policy для text/plain);
- нормалізація/обмеження image-вкладень під цільовий формат відображення.

## 3.3 Безпечний error contract

- валідаційні помилки не «розкривають» внутрішні деталі інфраструктури;
- REST і GraphQL мають передбачуваний формат помилок для UX та підтримки.

## 3.4 Що показати як доказ

- `docs/tz-compliance-checklist.md` (блоки XSS/captcha/attachments);
- create-comment walkthrough (`docs/code-walkthrough-create-comment.md`);
- test-evidence (`docs/code-walkthrough-test-evidence.md`).

## 4) Operations readiness — як пояснювати на захисті

## 4.1 Мінімальний story для production-like готовності

- є health/readiness на рівні сервісу;
- є smoke/go-no-go сценарій перевірки перед демонстрацією/релізом;
- є документовані артефакти (`docs/*`) для швидкого handoff і відтворюваності.

## 4.2 Що важливо підкреслити

1. **Контроль деградації**, а не тільки контроль «ідеального» режиму.
2. **Повторюваність перевірок**: той самий чекліст можна прогнати локально і в CI.
3. **Прозорість для рев’ю/аудиту**: рішення зафіксовані в walkthrough-документах.

## 5) Готовий скрипт відповіді (≈60 секунд)

«По нефункціональних вимогах ми закриваємо три площини.  
Перша — **продуктивність**: write-path тримаємо легким, а важкі інтеграції винесені в асинхронні канали; читання та пошук масштабуються через CQRS і пагінацію.  
Друга — **безпека**: перед створенням коментаря працюють captcha, валідація і sanitizer, а вкладення проходять strict policy по типах і розмірах.  
Третя — **операційна готовність**: є resilience-сценарії, health/smoke/go-no-go перевірки та документовані артефакти для повторюваного запуску.  
Тому система не лише функціонально коректна, а й передбачувана під навантаженням і збоями інтеграцій.»

## 6) FAQ (коротко)

1. **Що буде, якщо Elasticsearch недоступний?**  
   Пошук переходить на fallback, core-функція не блокується.

2. **Чому не робите всі side-effects синхронно?**  
   Це збільшить latency та крихкість write-path; асинхронна модель стійкіша.

3. **Як доводите безпеку вводу?**  
   Комбінуємо captcha + validation + sanitizer + attachment policy та показуємо це в test-evidence/checklist.

4. **Що з повторною доставкою message?**  
   Ідемпотентність забезпечується журналом `ProcessedMessages`.
