# Defense dry-run checklist (5–7 хв)

> Мета: зробити виступ перед захистом передбачуваним і повторюваним.

---

## 1) Структура виступу з таймінгом

1. **00:00–00:40 — Контекст і архітектура**
   - Що це за система.
   - Які шари і чому dependency rule важливий.

2. **00:40–02:20 — Основний сценарій create comment**
   - UI -> API -> Application -> DB.
   - Після commit: async події (MQ/SignalR/індексація).

3. **02:20–03:50 — Resilience та data consistency**
   - Idempotency через `ProcessedMessages`.
   - Eventual consistency між DB і search index.
   - Fallback пошуку при недоступному Elasticsearch.

4. **03:50–05:00 — NFR: security / performance / ops**
   - Captcha + sanitizer + attachment policy.
   - CQRS split + pagination + async side-effects.
   - Health/readiness і як приймається go/no-go.

5. **05:00–06:30 — Демо-кейси + висновок**
   - Degraded search mode.
   - Validation error з коректним UX-контрактом.

6. **06:30–07:00 — Q&A буфер**
   - 2–3 короткі відповіді на типові «складні» питання.

---

## 2) Обов’язкові демо-кейси

### Кейс A — Degraded search mode

- **Тригер:** симулюємо недоступність Elasticsearch.
- **Що показати:**
  - API не «падає» повністю;
  - пошук переходить у fallback;
  - UI зберігає робочий контракт відповіді.
- **Критерій успіху:** користувач отримує результат або контрольоване повідомлення без 500-хаосу.

### Кейс B — Validation UX contract

- **Тригер:** надсилаємо невалідний payload (порожній текст/некоректний email тощо).
- **Що показати:**
  - валідація спрацьовує на backend;
  - frontend відображає зрозумілу помилку;
  - немає «сирих» stack trace у UI.
- **Критерій успіху:** помилка прогнозована, формат стабільний, UX керований.

## 4) Швидкі відповіді на питання комісії

1. **Де транзакційна межа?**  
   У межах синхронного запису в БД; зовнішні інтеграції — асинхронно після успішного save.

2. **Як захищаєтесь від повторної доставки повідомлень?**  
   Через idempotency-журнал `ProcessedMessages` у consumer-пайплайні.

3. **Що якщо пошук недоступний?**  
   Система працює в degraded mode із fallback-пошуком, без зупинки базового сценарію.

4. **Чому це масштабовано?**  
   Через розділення відповідальностей (Clean/CQRS), подієвий fan-out і ізоляцію інтеграцій у Infrastructure.
